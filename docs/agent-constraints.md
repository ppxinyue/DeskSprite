
# DeskSprite — Agent Team 开发原则与执行约束

> **版本**：v1.0  
> **适用范围**：DeskSprite P0 阶段的所有 Agent 开发活动  
> **必须遵守**：本文档是所有 Agent 的行为准则，与 @docs/tech-spec-v0.2.md《DeskSprite 技术方案评审文档 v0.2》配套使用。遇冲突以本文档为准。

---

## 1. 核心原则

### 1.1 文档即真理
- 以@docs/tech-spec-v0.2.md《DeskSprite 技术方案评审文档 v0.2》为唯一权威来源。所有架构、模块边界、接口设计、功能清单均以该文档为准。
- 若发现文档存在模糊或矛盾之处，**不得自行解释或绕过**，必须记录到 `ISSUES.md` 并广播询问。
- 若因技术现实需要偏离文档，必须先更新文档对应章节（通过 PR 方式），经确认后方可执行。

### 1.2 透明可追溯
- 所有操作必须可解释。任何关键决策（技术取舍、方案变更、依赖选择）必须记录原因、考虑的替代方案和最终选择理由。
- 所有命令执行结果均保留日志，成功或失败均需记录。

### 1.3 最小权限
- 仅读写项目目录 `desksprite/` 下的文件。不访问系统其他目录，不读取用户个人文件（除非用户明确指定路径用于导出等功能）。
- 不访问环境变量中与项目无关的内容，不读取浏览器历史、系统偏好设置等。

### 1.4 安全优先
- 遇到任何可能造成不可逆后果的命令（删除、强制推送、修改系统配置等），必须先输出警告并请求确认。
- API Key 明文处理严格遵守文档 5.2 安全条款，不得在任何日志、错误信息、调试输出中打印明文 Key。

### 1.5 开发者指令优先
- 开发过程中，开发者（人类负责人）可能随时发出新的即时指令。该指令具有**最高优先级**。
- 即使新指令与 `tech-spec-v0.2.md` 内容不一致，或文档中未覆盖该指令，Agent 也必须**严格遵循开发者的最新指令**执行。
- 执行前，Agent 必须在 `PROGRESS.md` 中精确记录：
  - 指令内容
  - 发出者（开发者）
  - 影响到的任务/模块
  - 与现有文档的冲突点（如有）
- 执行完成后，在 `PROGRESS.md` 备注中标注执行结果。
- 若新指令涉及第 6 节中的**禁止指令**或潜在安全风险，Agent 应主动提醒开发者风险，并获得明确确认后方可执行。

## 2. 开发流程与进度管理

### 2.1 进度跟踪文件

项目初始化完成后，必须在项目根目录创建 `PROGRESS.md`，格式如下：

```markdown
# DeskSprite P0 开发进度

## 总体状态
- 开始时间：2026-xx-xx
- 当前阶段：P0
- 完成任务：x / 11 (A-K)
- 当前 Agent 分工：[Agent 1: 任务X] [Agent 2: 任务Y] [Agent 3: 任务Z]

## 任务进度

### A. 项目初始化
- 状态：✅ 完成 / 🔄 进行中 / ⏳ 待开始
- 负责人：Agent X
- 开始时间：
- 完成时间：
- 子任务：
  - [x] A1: 创建项目
  - [x] A2: 安装依赖
  - [ ] A3: 配置 Tailwind
  - [ ] ...
- 备注：[遇到的问题及解决方案]

### B. 数据库 + Keychain
- ...
```

**更新规则**：
- 开始一个任务时，将状态改为 🔄，记录开始时间。
- 每完成一个子任务，更新对应 checkbox。
- 整个任务完成时，将状态改为 ✅，记录完成时间和备注。
- 遇到阻塞时，在备注中说明阻塞原因和依赖方。

### 2.2 任务选择与声明

- 严格按照技术方案文档 6.1 节的依赖关系图选择任务。
- 只执行**当前依赖已全部满足**的任务。
- 开始执行前，在团队通信中广播：`Agent[X] 开始执行任务 [Y]：[任务名称]`。
- 同一任务不能被多个 Agent 同时执行（除非任务文档明确标注可拆分的子任务）。

### 2.3 并行开发隔离

- 不同 Agent 负责不同模块时，操作不同的文件目录：
  - Rust 命令按 `commands/` 下的文件名隔离（如 `keychain.rs`、`screenshot.rs` 分别由不同 Agent 负责）。
  - 前端按 `features/` 下的模块目录隔离（如 `features/ai/`、`features/pet/`、`features/chat/`）。
- 若两个 Agent 需要修改同一文件（如 `main.rs`、`tauri.conf.json`、`App.tsx`），必须：
  1. 先到者完成修改并提交。
  2. 后到者 `pull` 后合并，如有冲突必须手动解决并注释说明。

### 2.4 接口契约先行

- 任务 B（数据库）、G（AI Service）优先完成，产出 TypeScript 类型定义文件。
- 这些类型文件是任务 H、I 的接口契约。
- 若后续需要修改类型，必须在 `ISSUES.md` 中记录并通知受影响方。

## 3. 验证与质量标准

### 3.1 必须执行验证

- 技术方案文档中每个任务的“验证”栏**必须全部执行**，不得跳过。
- 验证结果记录在 `PROGRESS.md` 对应任务的备注中，格式：`验证通过：[简述如何验证的]`。

### 3.2 UI 相关需截图

以下任务的验证需要提供关键界面截图或录屏：
- 任务 F（灵宠动画）：灵宠在桌面显示、拖动、右键菜单
- 任务 H（悬浮对话框）：对话框展开、对话气泡、Markdown 渲染、全屏切换
- 任务 I（截图选区）：选区绘制、Token 预估、AI 分析结果
- 任务 E（设置中心）：设置页各面板

截图保存在 `docs/screenshots/` 目录下，按任务编号命名。

### 3.3 核心逻辑需测试

以下模块需编写单元测试或提供可运行的验证脚本：
- `aiService.ts`：至少覆盖 `streamChat`（模拟流式响应）和 `vision`（模拟图片响应）
- `petActivity.ts`：至少覆盖状态机转换和边界检测
- Keychain 命令：至少覆盖存取和脱敏返回

测试文件放在对应模块目录的 `__tests__/` 下。

## 4. Git 版本控制

### 4.1 提交粒度

- 每完成一个**任务**或一个**独立可运行的功能模块**，立即提交。
- **禁止**攒多个任务一起提交。

### 4.2 提交信息格式

```
<type>(<scope>): <简短描述>

[可选的详细说明]
```

- **type**：`feat`（新功能）、`fix`（修复）、`refactor`（重构）、`chore`（配置/工具）、`docs`（文档）、`test`（测试）
- **scope**：任务编号或模块名，如 `A`、`ai-service`、`pet-avatar`、`settings`、`database`

示例：
```
feat(database): 创建SQLite表结构和初始化迁移

完成 api_configs、system_prompts、conversations、messages、ai_usage_logs、settings 表的创建，
初始化猫十五默认 System Prompt。
```

### 4.3 分支策略

- `main`：受保护分支，仅通过 PR 合并。
- `p0/agent-{N}`：各 Agent 的开发分支，如 `p0/agent-1`。
- 每个任务完成后，从自己的分支向 `main` 发起 PR。
- PR 标题格式：`[P0] 完成任务 [X]：[任务名称]`。
- 合并前需要至少一个其他 Agent 的 Code Review（检查是否符合文档、是否有明显的安全问题）。

### 4.4 禁止操作

- **禁止** `git push --force` 或 `--force-with-lease` 到 `main` 分支。
- **禁止** `git commit --no-verify` 绕过检查。
- **禁止** 提交包含明文 API Key 或密钥的代码。

## 5. 问题处理与经验总结

### 5.1 问题记录文件

项目根目录创建 `ISSUES.md`，格式如下：

```markdown
# DeskSprite 开发问题记录

## ISSUE-001
- 发现时间：2026-xx-xx
- 发现者：Agent X
- 相关任务：B. 数据库
- 严重程度：阻塞 / 影响进度 / 已解决
- 问题现象：[描述遇到的问题]
- 原因分析：[为什么会发生]
- 解决方案：[怎么解决的]
- 经验总结：[下次如何避免]
- 是否需更新技术文档：是 / 否
```

### 5.2 问题升级

- 遇到**阻塞性问题**（无法继续当前任务），必须：
  1. 记录到 `ISSUES.md`。
  2. 广播通知其他 Agent。
  3. 暂停当前任务，等待决策。
- 禁止在无解释的情况下反复尝试同一方案超过 3 次。

### 5.3 事后总结

- 每个任务完成后，在 `PROGRESS.md` 的备注中写一段简短的“经验要点”（1-3 句话），总结本任务的关键发现和注意事项。
- P0 全部完成后，整理 `ISSUES.md` 作为项目知识资产。

## 6. 权限与安全约束

### 6.1 授予权限

Agent 在项目目录 `desksprite/` 内拥有以下权限：
- 创建、读取、修改、删除项目内所有文件
- 执行 `cargo`、`pnpm`、`npm`、`npx`、`git`、`tauri-cli` 等开发工具命令
- 创建临时文件和目录（如截图测试用的临时图片）
- 读取系统信息（如桌面分辨率，用于开发调试）

### 6.2 禁止指令清单

以下命令**绝对禁止**执行。Agent 遇到此类指令时必须拒绝并发出警告：

**文件系统危险操作**：
- `rm -rf /` 或任何以系统根目录为目标的删除命令
- `rm -rf ~` 或任何以用户主目录为目标的删除命令
- 删除项目外任何文件
- 修改 `/etc/`、`/System/`、`C:\Windows\`、`C:\Program Files\` 等系统目录内容

**Git 危险操作**：
- `git push --force` 到 `main` 分支
- `git reset --hard HEAD~n` 丢弃已推送的提交（除非用户明确要求且已备份）
- `git commit --no-verify` 绕过 hooks

**系统级危险操作**：
- `sudo` 或管理员权限下的任何命令（除非用户明确要求安装系统级依赖）
- 修改系统 PATH、环境变量
- 修改防火墙规则、hosts 文件
- 安装或卸载系统级软件包

**隐私与安全红线**：
- 读取 `~/.ssh/`、`~/.gnupg/` 等密钥目录
- 读取浏览器 cookie、密码管理器数据
- 将任何文件上传到外部服务器（除 LLM API 调用外）
- 在任何日志、提交、注释中打印 API Key 明文

### 6.3 模糊指令处理

- 若用户给出的指令可能触发上述禁止项，Agent 应：
  1. 输出警告，说明风险。
  2. 提议安全替代方案。
  3. 等待用户确认后再执行。

## 7. 多 Agent 协作与通信

### 7.1 通信约定

**任务广播**（在团队频道中）：
- 开始任务：`[Agent-1] 开始执行 任务 F：灵宠动画+拖动`
- 完成任务：`[Agent-1] 完成任务 F：灵宠动画+拖动。产出：PetAvatar.tsx, petStore.ts, animations.ts。验证通过。`
- 遇到阻塞：`[Agent-2] 阻塞：任务 H 等待 任务 G 的 AiService 类型定义。`

**接口变更通知**：
- 若发现需要修改与其他 Agent 共享的类型定义或接口签名，必须：
  1. 先通知受影响方。
  2. 在单独的一次提交中完成修改，提交信息标注 `BREAKING`。
  3. 在 `ISSUES.md` 中记录变更原因。

### 7.2 同步检查点

在以下节点，所有 Agent 暂停新任务，进行同步检查：
- **检查点 1**：任务 A 全部完成后（确认项目基础统一）
- **检查点 2**：任务 B/C/D 全部完成后（确认数据库、窗口、组件库就绪）
- **检查点 3**：任务 E/F/G 全部完成后（确认设置、动画、AI 服务就绪，接口对齐）
- **检查点 4**：任务 H/I/J 全部完成后（集成前最后检查）

每个检查点需确认：
- 各 Agent 的产出是否兼容。
- 共享类型/接口是否一致。
- `PROGRESS.md` 状态是否统一。
